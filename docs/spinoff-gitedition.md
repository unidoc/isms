# GitEdition.com — Spinoff Plan

## What It Is

GitEdition is a standalone document platform built from the strongest part of `isms.sh`:

- git-backed markdown documents
- round-based review
- formal publish workflow
- versioned history
- white-label delivery
- self-hosted or cloud deployment

It is **not** a compliance product.
It is a general publishing and review system for teams that need documents to move cleanly from rough working notes to trusted published content.

Examples:

- internal knowledge bases
- policies and procedures
- legal drafts and contract playbooks
- consulting deliverables
- client handbooks
- engineering documentation
- team wikis
- private note systems that can turn into structured content later

---

## Core Thesis

Most tools split the world badly:

- note apps are good for raw capture but weak for formal review and publishing
- wiki tools are okay for browsing but weak for approval and audit trail
- Google Docs is decent for ad hoc collaboration but weak for versioned truth
- enterprise knowledge tools become bloated and rigid fast

GitEdition should win because it joins these worlds cleanly:

1. **Write rough content fast**
2. **Review it properly when it matters**
3. **Publish it cleanly when it becomes official**
4. **Keep the whole history in one place**

That is the product.

Not just “wiki with AI”.
Not just “docs with git”.
Not just “Google Docs but self-hosted”.

It is:

**private notes -> collaborative drafts -> reviewed editions -> published knowledge**

---

## Unique Selling Point

The unique selling point is not one feature.
It is the combination of four things that are rarely strong in the same system:

### 1. Real Working Drafts

People can write messy, partial, evolving content without pretending it is already official.

### 2. Real Review

When a document matters, it goes through actual review:

- rounds
- this round vs all changes
- inline suggestions
- proposed revision
- immutable decision trail

This is much stronger than comments-only workflows.

### 3. Real Publishing

A document can become:

- private
- shared
- internal
- public

with a clean published version, branded delivery, and stable URLs.

### 4. Real History

The same system keeps:

- authoring history
- review history
- publish history
- AI assistance history

That is the bomb.

The strongest one-line positioning is:

**Write like a notebook, review like a law firm, publish like a real product.**

Alternative line:

**From rough notes to trusted editions.**

## Content Lifecycle

This should be the heart of the product.

### 1. Private

Personal notes, rough drafts, scratchpads, research, diary-style writing, early ideas.

This is where someone writes:

- half-formed ideas
- meeting notes
- raw observations
- draft paragraphs
- private working material

No pressure for structure yet.

### 2. Shared

A document or notebook can be shared with selected people for input.

This is lighter than publishing.
It is just:

- let others read
- optionally let others edit
- optionally let others comment

### 3. Internal

This is trusted organizational content.

Examples:

- internal wiki pages
- handbooks
- procedures
- team documentation
- approved internal policies

### 4. Public

This is published outward-facing content.

Examples:

- public docs
- customer knowledge base
- partner handbook
- public policy page
- documentation site

The same engine should support all four.

That is much more powerful than “everything is just a wiki page”.

## Access And Publishing Model

This needs to be explicit.

Each document should be able to live in one of these visibility states:

- `private`
- `shared`
- `internal`
- `public`

### Private

Visible only to the owner, or owner plus admins.

Good for:

- notebook entries
- early drafts
- personal writing

### Shared

Visible to selected people or groups.
Optional edit permission.

Good for:

- collaborative drafts
- client review
- guest reviewer flows

### Internal

Visible to authenticated workspace members according to workspace or folder rules.

Good for:

- team docs
- knowledge base
- internal handbook

### Public

Published without login.
Can be branded and served on custom domain.

Good for:

- external docs
- help center
- public policy site
- public handbook

This visibility model is much stronger than just:

- private doc
- public site

It gives GitEdition a full content spectrum.

A document can exist as:

- working draft
- reviewed draft
- published edition

Important rule:

This is where GitEdition beats normal wiki tools.

all on the same engine.

### Lightweight Collaboration

For fast team writing:

- comments
- mentions
- shared editing
- quick iteration

### Formal Collaboration

For documents that need authority:

- send for review
- review rounds
- suggestions

This dual mode is important.

It lets the product serve both:

- normal team docs
- high-stakes docs

without forcing the heavy workflow on everything.

## Technical Approach

### Build Strategy

- same Go binary and Vue app foundation
- feature-flag or build-tag removal of ISMS modules
- shared core for auth, git store, review engine, branding, API
- separate product shell, landing page, templates, and publishing features

### Repository Strategy

1. start in monorepo with build tags or product flags
2. split shared core later only if necessary

That keeps early momentum high.

## Pricing Direction

Possible shape:

- **Free / OSS**: self-hosted core, serious but minimal polish
- **Cloud**: easiest hosted experience
- **Business**: white-label, SSO, advanced permissions, publishing controls
- **Enterprise**: AI workflows, richer review controls, premium integrations, premium support

The hosted product should win on convenience and intelligence.
